# C++面试Agent 产品技术规格文档

> 版本：v1.0 | 日期：2026-05-05 | 状态：待开发

---

## 1. 需求概述

### 1.1 用户画像

| 维度 | 描述 |
|------|------|
| 目标群体 | 校招应届生 |
| 学历背景 | 本科/硕士在校生，计算机或相关专业 |
| C++基础 | 学过C++课程但缺乏实战深度，对底层原理理解不扎实 |
| 痛点 | 面试知识点碎片化、无法系统组织、对底层原理一知半解、面试复盘无参考标准 |
| 使用场景 | 日常系统性学习C++面试知识点；面试后复盘提分 |
| 期望 | 循序渐进掌握知识点，面试时能清晰、准确、有逻辑地回答 |

### 1.2 产品定位

**系统性掌握知识点** —— 帮助校招应届生从零构建C++面试知识体系，通过"知识点定位→核心原理拆解→常见考法说明→易错点提示"的引导式学习路径，实现从"背答案"到"理解原理"的转变。

### 1.3 核心目标

| 目标 | 量化指标 |
|------|----------|
| 知识覆盖 | 覆盖C++全栈面试6大方向（核心语法、STL、操作系统、网络、数据库、设计模式）|
| 问答质量 | 简要回答≤200字、详细回答结构化4段式、知识库匹配回答经双重校验 |
| 复盘深度 | 5维度评估（准确性/完整性/逻辑性/专业术语使用/匹配度），每维度分点指出不足 |
| 响应性能 | 简答≤5s、详答≤15s、复盘报告≤60s |
| 并发能力 | ≥10 QPS |

## 2. 功能详细设计

### 2.1 LangGraph框架适配性评估

#### 评估结论：✅ 适配，推荐采用

| 评估维度 | 需求 | LangGraph能力 | 适配度 |
|----------|------|---------------|--------|
| 多步骤流程编排 | 功能1需要RAG检索→阈值判断→分支生成→格式化；功能2需要文本清洗→问题提取→逐题解答→对比评估→报告生成 | 基于有向图的状态机，支持条件边（conditional_edge）、并行节点（Send API）、子图嵌套 | ★★★★★ |
| 状态管理 | 功能2需在多节点间传递面试问题列表、逐题解答结果、评估结果等中间状态 | 内置TypedDict状态管理，支持checkpoint持久化 | ★★★★★ |
| 分支逻辑 | 检索相似度≥0.8走知识库回答，<0.8走在线搜索→LLM生成 | conditional_edge原生支持，零额外开发 | ★★★★★ |
| 并行处理 | 复盘场景对N个问题并行解答 | LangGraph Send API支持动态并行节点 | ★★★★☆ |
| 流式输出 | 分级响应需流式返回 | 原生支持stream/stream_events | ★★★★★ |
| Docker部署 | 容器化部署 | 纯Python库，无特殊依赖 | ★★★★★ |
| 极低预算 | 开源免费 | Apache 2.0开源协议 | ★★★★★ |
| 模型可配置 | .env配置多模型API | 原生支持ChatOpenAI及 LangChain Chat模型生态 | ★★★★★ |
| 10 QPS并发 | 小规模并发 | 无状态设计，配合FastAPI + uvicorn worker水平扩展 | ★★★★☆ |

**潜在风险与应对：**
- LangGraph checkpoint在10 QPS下可能产生I/O瓶颈 → 默认使用内存checkpoint，生产环境可选SQLite/PostgreSQL
- 子图嵌套层数过深导致调试困难 → 限制子图嵌套≤2层，关键节点添加详细日志

### 2.2 功能1：单个面试问题解答

#### 2.2.1 执行流程

```
用户输入面试问题
    │
    ▼
[Node: 对齐与改写] ─── 对用户口语化提问进行语义理解，提取核心知识点关键词
    │
    ▼
[Node: RAG检索] ─── 用关键词+原始问题做多路检索，获取Top-K结果
    │
    ▼
[Conditional Edge: 阈值判断] ─── 最高相似度是否 ≥ 0.8？
    │                         │
   YES                       NO
    │                         │
    ▼                         ▼
[Node: 校验知识库内容]      [Node: 在线搜索]
  - 知识点准确性校验         - 构造搜索查询
  - 时效性校验               - Tavily/SearXNG搜索
    │                         - 抓取Top-3正文
    │                         │
   校验通过？                 │
    │      │                  │
   YES    NO ──→ [Node: LLM生成回答] ←── 搜索结果作为参考上下文
    │                │
    ▼                ▼
[Node: 格式化输出]
  ├─ 简要回答（≤200字）
  └─ 详细回答（4段式结构）
  + 来源标记：知识库 / 在线搜索 / AI生成
    │
    ▼
返回结果
```

#### 2.2.2 输出规则

**简要回答（short_answer）：**
- 字数：≤200字
- 内容要求：覆盖核心结论 + 关键判定标准/核心特性
- 风格：口语化作答逻辑，贴合面试场景，无冗余内容
- 禁止：代码示例、延伸扩展、铺垫性介绍

**详细回答（detailed_answer）：** 采用固定4段式结构：

| 段落 | 标题 | 内容要求 |
|------|------|----------|
| 第1段 | 📍 知识点定位 | 用1-2句话定位该知识点在C++知识体系中的位置，说明"这是什么" |
| 第2段 | 🔍 核心原理拆解 | 逐步拆解底层原理，从概念定义→机制说明→关键细节，用类比/对比辅助理解，必要时辅以代码示例（<30行） |
| 第3段 | 📝 常见考法说明 | 列举2-3种面试常见问法变体及其答题要点，如"手写实现"、"与XX的区别"、"底层原理追问" |
| 第4段 | ⚠️ 易错点提示 | 列出2-3个考生高频出错的点，如"未考虑XX边界情况"、"混淆XX和XX的概念" |

**回答来源标记：**
- 知识库回答：标注 `[来源: 知识库]` + 具体出处（如"《Effective C++》Item 11"）
- 在线搜索回答：标注 `[来源: 在线搜索]` + 搜索URL来源
- LLM生成回答：标注 `[来源: AI生成]` + 置信度等级（高/中/低）

#### 2.2.3 异常处理逻辑

| 异常场景 | 处理策略 |
|----------|----------|
| 问题与C++无关 | 返回提示："该问题不在C++面试知识范围内，请提出C++相关的面试问题。" |
| 问题过于模糊 | 返回追问："您的问题比较宽泛，请问您具体想了解的是？例如：①XXX ②XXX ③XXX" |
| RAG检索服务不可用 | 降级为在线搜索→LLM生成，日志记录告警，回答标注`[来源: AI生成（知识库暂不可用）]` |
| 在线搜索服务不可用 | 跳过在线搜索，直接LLM生成，回答标注`[来源: AI生成（在线搜索暂不可用）]` |
| 在线搜索超时（>5s） | 跳过在线搜索，直接LLM生成，日志记录超时 |
| LLM API调用失败 | 返回错误码503，提示"服务暂时不可用，请稍后重试"，支持自动重试1次 |
| LLM生成内容被安全审核拦截 | 返回提示："该问题涉及敏感内容，请调整问题后重试。" |
| 知识库内容校验不通过（准确性或时效性） | 降级为LLM生成，日志记录校验失败原因 |
| 生成超时 | 简答5s/详答15s超时后返回已生成的部分内容 + "[回答因超时截断]" |

### 2.3 功能2：面试复盘

#### 2.3.1 执行流程

```
用户输入面试录音转写文本（长文本）
    │
    ▼
[Node: 文本预处理]
  ├─ 口语化内容规整（去除"嗯/啊/就是/然后"等语气词）
  ├─ 噪音内容过滤（去除无关闲聊、重复内容）
  ├─ 说话人分离（区分面试官/面试者发言）
  └─ 输出：结构化对话文本
    │
    ▼
[Node: 问题提取]
  ├─ 从面试官发言中提取每个独立问题
  ├─ 合并追问/补充说明为主问题的子问题
  └─ 输出：问题列表 [{question_id, question_text, sub_questions}]
    │
    ▼
[Node: 逐题解答] ─── 对每个问题并行执行功能1的完整流程（检索→阈值判断→生成）
  ├─ 问题1 → 参考答案1
  ├─ 问题2 → 参考答案2
  └─ ...
    │
    ▼
[Node: 回答提取]
  ├─ 从面试者发言中提取每个问题的实际回答
  └─ 输出：回答列表 [{question_id, actual_answer}]
    │
    ▼
[Node: 逐题对比评估] ─── 对每个问题对比面试者实际回答 vs 参考答案
  ├─ 准确性评估：关键知识点是否正确
  ├─ 完整性评估：是否覆盖所有要点
  ├─ 逻辑性评估：回答是否有条理
  ├─ 专业术语使用评估：是否使用了准确的专业术语
  └─ 匹配度评估：回答是否切题
  ── 每维度：分点指出不足 + 针对性优化建议
    │
    ▼
[Node: 报告生成]
  ├─ 模块1：面试问题汇总
  ├─ 模块2：各问题参考答案
  ├─ 模块3：各问题回答评估
  ├─ 模块4：整体面试表现总结
  └─ 模块5：提升建议
    │
    ▼
返回结构化复盘报告
```

#### 2.3.2 输出规则

**复盘报告结构（严格遵循5大模块）：**

**模块1：面试问题汇总**
```
📋 面试问题汇总
────────────────
共 N 个问题：

Q1: [问题文本]
  - 追问1: [追问文本]
  - 追问2: [追问文本]

Q2: [问题文本]
...
```

**模块2：各问题参考答案**
```
📚 各问题参考答案
────────────────

【Q1】[问题文本]
  简要回答：[≤200字]
  详细回答：
    📍 知识点定位：...
    🔍 核心原理拆解：...
    📝 常见考法说明：...
    ⚠️ 易错点提示：...

【Q2】[问题文本]
...
```

**模块3：各问题回答评估**
```
📊 各问题回答评估
────────────────

【Q1】[问题文本]
  面试者回答摘要：[1-2句话概括]

  | 评估维度 | 评分(1-5) | 不足之处 | 优化建议 |
  |----------|-----------|----------|----------|
  | 准确性   | X/5       | 分点说明  | 具体建议  |
  | 完整性   | X/5       | 分点说明  | 具体建议  |
  | 逻辑性   | X/5       | 分点说明  | 具体建议  |
  | 专业术语 | X/5       | 分点说明  | 具体建议  |
  | 匹配度   | X/5       | 分点说明  | 具体建议  |

  综合评分：X/5

【Q2】...
```

**模块4：整体面试表现总结**
```
🎯 整体面试表现总结
────────────────
- 总体评分：X/5
- 表现较好的方面：[列出2-3个 strengths]
- 需要重点提升的方面：[列出3-5个 weaknesses]
- 知识点掌握分布：[按6大方向统计正确率]
- 回答风格特点：[如"回答偏简略"、"逻辑跳跃"等]
```

**模块5：提升建议**
```
💡 提升建议
────────────────
1. 优先补强方向：[根据评估得分最低的2-3个知识方向]
   - 方向A：建议学习 [具体知识点]，参考 [资料推荐]
   - 方向B：...

2. 回答技巧提升：
   - [针对逻辑性不足的具体建议]
   - [针对专业术语使用的具体建议]

3. 建议练习题目：[3-5道关联面试题推荐]

4. 学习路径建议：[按优先级排列的下一步学习计划]
```

#### 2.3.3 异常处理逻辑

| 异常场景 | 处理策略 |
|----------|----------|
| 输入文本为空或过短（<50字） | 返回错误："输入文本过短，请提供完整的面试录音转写文本。" |
| 无法识别面试官/面试者 | 尝试基于语义区分提问者与回答者；若仍失败，提示用户标注说话人 |
| 问题提取结果为0 | 返回提示："未识别到面试问题，请确认输入为面试对话文本。" |
| 单个问题解答超时 | 设单题解答上限10s，超时则该题参考答案标注"[生成超时]"，继续处理下一题 |
| 部分问题解答失败 | 跳过失败问题，报告中标注"[该题解答失败]"，其他问题正常输出 |
| 输入文本超过token限制 | 分段处理，每段≤4000 token，合并结果后生成报告 |
| 整体报告生成超时 | 60s超时后返回已生成的部分模块 + "[报告因超时截断，请重试获取完整报告]" |

**用户自定义材料指引：**
- 用户将文件放入 `./book/` 任意子目录即可
- 新增文件后需执行 `python scripts/seed_knowledge.py --rebuild` 重建索引
- 建议在文件名或目录名中体现知识分类（如 `stl/`、`network/`），便于溯源
- Markdown文件建议规范使用标题层级，一级标题(##)为知识大类，二级标题(###)为具体知识点

#### 5.1.3 文档Chunk切分策略

采用基于Markdown标题层级的切分策略，适配任意格式文档（非结构化文档、教程、书籍章节等），不限于QA对格式。

**切分流程：**

1. **格式统一转换**：通过 [MarkItDown](https://github.com/microsoft/markitdown)（微软开源）将非Markdown文件统一转为Markdown格式
2. **标题层级切分**：按Markdown标题标记（`#`/`##`/`###`）对文档进行层次化切分
3. **语义完整性保障**：每个chunk保持在单个标题段落内，不跨标题切分

**切分参数：**

| 参数 | 设定值 | 说明 |
|------|--------|------|
| 切分依据 | Markdown标题层级 | 以 `##`（二级标题）为首选切分边界，`###`（三级标题）为辅助切分边界 |
| 最小切分单位 | `###` 三级标题段落 | 三级标题下的内容作为一个chunk，不进一步切分 |
| chunk_size上限 | 800 tokens | 超过上限的chunk按段落换行符二次切分 |
| chunk_size下限 | 50 tokens | 低于下限的chunk合并至上级标题chunk中 |
| chunk_overlap | 0 tokens | 标题层级切分天然保留上下文，无需重叠 |
| 父子关联 | 保留 | 每个chunk记录其父标题（上级heading），检索时可回溯上下文 |

**元数据字段：**

| 字段 | 来源 | 说明 |
|------|------|------|
| `source` | 文件路径 | 原始文件相对路径（如 `book/STL源码剖析.md`） |
| `heading_level` | 自动提取 | 标题层级（2或3） |
| `heading_text` | 自动提取 | 标题文本（如"vector底层实现"），用于关键词匹配 |
| `parent_heading` | 自动提取 | 父级标题文本，用于上下文回溯 |
| `category` | LLM推断 | 基于标题内容自动推断知识分类（C++核心语法/STL/操作系统/网络/数据库/设计模式） |
| `sub_category` | LLM推断 | 细分方向（如"面向对象-虚函数"） |
| `difficulty` | LLM推断 | 难度等级（基础/中等/进阶） |
| `cxx_standard` | 内容推断 | 涉及的C++标准版本标注 |
| `last_verified` | 入库时间 | 入库时自动标注当前日期 |
| `tags` | 合并 | 标题+内容中提取的技术关键词 |

**切分示例：**

原始Markdown：
```markdown
## STL容器

### vector底层实现
vector是动态数组，底层采用连续内存空间...
扩容机制：当容量不足时，以2倍（或1.5倍）扩容...

### map与unordered_map
map基于红黑树，有序，查找O(logN)...
unordered_map基于哈希表，无序，查找平均O(1)...
```

切分结果：
- chunk1: `{heading_text: "vector底层实现", parent_heading: "STL容器", content: "vector是动态数组...扩容机制..."}`
- chunk2: `{heading_text: "map与unordered_map", parent_heading: "STL容器", content: "map基于红黑树...unordered_map基于哈希表..."}`

#### 5.1.4 知识库更新策略

- 更新频率：季度更新
- 更新流程：
  1. 用户在 `./book/` 目录中新增/替换文档
  2. 执行 `python scripts/seed_knowledge.py --rebuild` 重建索引
  3. 系统自动检测新增文件（基于文件hash去重）

### 5.2 检索方案

#### 5.2.1 多路检索策略

```
用户问题
    │
    ├──→ 路径1: 关键词检索（BM25风格）
    │    - 提取问题中的技术关键词
    │    - 在元数据category/sub_category/heading_text中精确匹配
    │
    ├──→ 路径2: 语义向量检索
    │    - 问题经text2vec-large-chinese向量化
    │    - 在Chroma中做cosine相似度检索Top-5
    │
    └──→ 合并去重 → 按相似度排序 → 取Top-5
              │
              ├── max_score ≥ 0.8 → 知识库回答
              └── max_score < 0.8 → 路径3: 在线搜索
                                     - Tavily/SearXNG搜索
                                     - 优选参考资源列表中的站点
                                     - 抓取Top-3正文 → LLM基于搜索结果回答
```

#### 5.2.2 阈值判断与双重校验

**阈值判断逻辑：**
- 对合并后的Top-5结果，取最高相似度分数 `max_score`
- `max_score ≥ 0.8`：进入知识库内容校验流程
- `max_score < 0.8`：进入在线搜索流程，搜索结果作为LLM参考上下文
- 在线搜索失败/超时：降级为纯LLM生成

**双重校验流程：**

| 校验维度 | 校验方法 | 不通过处理 |
|----------|----------|-----------|
| 知识点准确性 | LLM校验：用小参数模型/低成本模型对知识库内容做事实性审核（Prompt: "判断以下C++知识点描述是否准确，仅回答是/否及原因"） | 降级为LLM生成完整回答 |
| 时效性 | 规则校验：检查`last_verified`是否在6个月内 + `cxx_standard`是否覆盖C++11/14/17/20相关标注 | 超期条目仍可使用但标注"[该内容最近验证于XXXX-XX，请注意时效性]"，涉及已废弃标准的条目降级 |

**双重校验成本控制：**
- 使用最便宜的模型API（如DeepSeek-Chat）做准确性校验
- 时效性校验为纯规则判断，零成本
- 校验Prompt精简至<200 token

### 5.3 在线搜索补充方案

当知识库检索最高相似度 < 0.8 时，在降级为纯LLM生成之前，增加在线搜索环节，以获取最新的技术资料作为LLM回答的参考上下文。

#### 5.3.1 搜索流程

```
知识库检索 max_score < 0.8
    │
    ▼
[Node: 在线搜索]
  ├─ 构造搜索查询（基于rewrite后的关键词）
  ├─ 调用搜索引擎API获取Top-5结果
  ├─ 抓取结果页面正文内容（限前3条，每条≤2000字）
  └─ 输出：搜索结果文本
    │
    ▼
[Node: LLM基于搜索结果生成回答]
  ├─ 将搜索结果作为参考上下文注入Prompt
  └─ 标注来源: [来源: 在线搜索] + 搜索URL
```

#### 5.3.2 搜索策略

| 配置项 | 方案 |
|--------|------|
| 搜索引擎 | Tavily Search API（专为LLM优化的搜索API，返回清洗后的内容） |
| 备选方案 | SearXNG（自建开源元搜索引擎，零成本） |
| 搜索查询构造 | 使用rewrite节点输出的关键词组合为搜索query |
| 结果数量 | Top-5搜索链接，抓取前3条正文 |
| 内容长度限制 | 每条≤2000字，总计≤4000字，避免超出LLM上下文 |
| 超时控制 | 搜索总耗时≤5s，超时则跳过在线搜索，直接LLM生成 |
| 结果过滤 | 优先来自以下在线参考资源列表中的站点 |

#### 5.3.3 在线搜索回退链路

```
知识库匹配 ≥ 0.8  →  使用知识库回答
知识库匹配 < 0.8  →  在线搜索  →  搜索成功  →  LLM基于搜索结果回答 [来源: 在线搜索]
```

```json
{
```

### C. LangGraph State定义

```python
# 功能1 State
class QuestionState(TypedDict):
    question: str
    rewritten_query: str
    keywords: list[str]
    retrieval_results: list[dict]
    max_similarity: float
    use_knowledge_base: bool
    validation_passed: bool
    knowledge_base_content: str | None
    knowledge_base_metadata: dict | None
    online_search_results: str | None
    online_search_urls: list[str]
    short_answer: str
    detailed_answer: dict
    source_info: dict
    error: str | None

# 功能2 State
class ReviewState(TypedDict):
    raw_transcript: str
    metadata: dict | None
    cleaned_transcript: str
    questions: list[dict]
    interviewee_answers: list[dict]
    reference_answers: list[dict]
    evaluations: list[dict]
    report: dict
    error: str | None
```
